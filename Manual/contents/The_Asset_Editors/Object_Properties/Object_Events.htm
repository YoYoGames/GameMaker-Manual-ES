Eventos de objetos

Entonces, ¿qué son los eventos de object? Básicamente, son momentos discretos en el juego loop en los que se hace que ocurran cosas en función de lo que hayas programado para ellos. GameMaker trabaja con ciclos de estos eventos - desde el momento en que un room se inicia hasta el momento en que se termina hay un juego loop en marcha donde cada paso una serie de eventos se ejecutan o se comprueban, y usted puede optar por colocar el código o GML Acciones visuales en su objects que responden a estos eventos (un paso es un momento en el tiempo de juego, gobernado por el room ajuste de velocidad, y también puede ser llamado un marco) .

Estas opciones son:

  • Añadir evento - Añadir un nuevo evento de la lista de eventos.
  • Cortar / Copiar / Pegar Eventos - Cortar, copiar o pegar el evento seleccionado. También puede utilizar los atajos de teclado estándar: / + X, / + C, y / + V.
  • Duplicar Evento - Duplica el evento elegido. Esto tomará el contenido del evento elegido y lo duplicará en otro evento que usted especifique.
  • Cambiar Evento - Cambiar el evento. Esto le pedirá que elija otra categoría de evento y, una vez seleccionado, el contenido del evento actual se cambiará al nuevo evento (eliminando el evento original en el proceso).
  • Convertir a GML Visual / Código - Los eventos de Object se crearán utilizando el método elegido al crear el proyecto, ya sea mediante código o mediante acciones de GML Visual. Con esta opción puede convertir de uno a otro, independientemente del tipo de proyecto.
  • Borrar Evento - Borrar el evento (o eventos).

Al eliminar eventos, puede utilizar + para seleccionar varios eventos y luego eliminarlos todos juntos.Cada object que cree tiene su propia lista discreta de eventos que se añaden a ella desde el Editor de Objetos. Estos eventos se dividen en dos categorías:

  • los que ejecutan cada paso del juego
  • las que se "disparan" por un evento del juego, como que la instancia llegue al borde de room o que se pulse el teclado o el ratón.

La lista completa de eventos se presenta a continuación:

Crear

Este evento ocurre cuando se crea por primera vez una instancia de object, y es lo primero que ocurre dentro de una instancia colocada en room a través del Editor de Room cuando se introduce un room. Esto significa que este evento es el lugar ideal para inicializarvariables, iniciar Líneas de Tiempo, establecer Rutas, etc... y hacer cualquier otra cosa que generalmente sólo necesita ser hecha una vez o sólo cuando una instancia es creada por primera vez en el room. Si su object tiene cualquier Variable de Objeto o Variable de Instancia añadida en el Editor Object o en el Editor Room, entonces estas variables serán inicializadas primero y luego se ejecutará el Evento Crear.

Recuerde que puede modificar cualquier cosa que configure en el Evento de Creación desde el Código de Creación de Instancia en el Editor Room, ya que se ejecuta directamente después del evento de creación de la instancia y puede ser utilizado para crear Variables de Instancia o para anular cualquier variable añadida como Object Variables o en el Evento de Creación real.

Destruye

Este evento es el que se ejecuta cuando se destruye una instancia. A menudo se pasa por alto cuando se añaden comportamientos a objects, pero puede ser muy útil, por ejemplo creando efectos de explosión o de partículas cuando se mata a un enemigo, o para reaparecer una nueva instancia de object en otra parte de room, o incluso para añadir puntos a una puntuación.

Limpieza

Este evento será llamado después de cualquier evento que elimine una instancia de la object de la room. Así, se activará si:

  • la instancia se destruye
  • el room termina
  • el juego termina

Está diseñado para que lo utilices para "limpiar" los recursos dinámicos que puedas tener en tu juego (como superficies, data structures, etc...) o para realizar cualquier tarea que necesites que se realice una vez que la instancia sea eliminada del juego de cualquier manera.

Ten en cuenta que este evento será llamado instantáneamente después del evento que lo activó, pero la instancia no será realmente eliminada del juego hasta el final del evento actual. Por ejemplo, si llamas a instance_destroy() en el Evento de Paso, entonces el Evento de Destrucción será llamado, luego el Evento de Limpieza, y luego el resto del Evento de Paso terminará de ejecutarse. Esto significa que cualquier código que tengas después de la llamada a instance_destroy() se seguirá ejecutando y será una causa potencial de errores si has limpiado una estructura de datos o algún otro recurso que el código requiera, por lo que hay que tener cuidado al usar este evento.

Alarma

La categoría de alarmas está dividida en 12 eventos, uno por cada una de las posibles alarmas que se pueden establecer en una instancia. Así, al hacer clic en la categoría Añadir alarma, se presenta esta ventana:

Aquí seleccionas la alarma que deseas crear y, una vez hecho esto, verás que se ha añadido a la ventana de eventos permitiéndote añadirle código de forma normal. ¿Pero qué es una alarma? Bueno, es un evento especial que no hace nada a menos que la alarma haya sido configurada previamente, y entonces esperará hasta que esa alarma haya contado hasta 0 antes de ejecutar las acciones o el código que hayas añadido en ella.

Una vez que la alarma ha llegado a 0 y ejecuta el código, entonces contará hacia abajo a -1, donde permanecerá hasta que se establezca de nuevo (lo que significa que usted puede comprobar el valor de una alarma para ver si es mayor que -1, que le dirá si se está ejecutando o no). Así que, digamos que usted establece alarm[0] en el evento de creación de la object a 30, esto significa que GameMaker contará 30 pasos de juego antes de ejecutar las acciones o el código que se colocan en el evento alarm[0]. Tenga en cuenta que el establecimiento de una alarma a 0 no se ejecutará el código de la alarma, ya que el evento se activa, pero la alarma se establece en -1 inmediatamente, por lo que el código se omite. Si necesitas que una alarma ejecute el siguiente paso, entonces debes ponerla a 1.

Esto puede ser muy útil ya que te permite poner cosas en movimiento en momentos precisos, e incluso puedes hacer que se repitan ya que no hay nada que te impida poner una alarma en su propio evento. Imagina que tienes un monstruo y quieres que gire a la derecha cada tres segundos... pues bien, establecerías una alarma en su evento de creación a la velocidad de room * 3 (si la velocidad de room es 30, eso son 30 pasos por segundo, así que multiplícalo por 3 y tendrás 3 segundos) y luego en el evento de alarma tendrías el código o la acción para establecer su dirección, así como la acción (o código) para establecer su alarma a la velocidad de room * 3 de nuevo. De esta manera, puedes configurar un juego simple loops donde las cosas sólo suceden en intervalos específicos.

Cabe destacar que una alarma sin acciones o código en ella no tendrá cuenta atrás. Sin embargo, incluso con sólo un comentario y sin código o acciones, entonces la alarma continuará la cuenta atrás y se puede establecer y comprobar como lo haría normalmente.

Paso

GameMaker divide el tiempo en pasos con la velocidad de room definiendo cuántos de estos pasos se supone que hay por segundo (un paso también puede ser llamado un cuadro). Un solo paso, es básicamente el loop que se ejecuta constantemente con todos los eventos que se comprueban y desencadenan como sea necesario mientras el juego se ejecuta, así que como se puede imaginar, el evento de paso es un evento que se comprueba cada paso del juego mientras la instancia existe.

El evento del paso se compone en realidad de tres sub eventos que se describen a continuación:

Para la mayoría de las cosas el evento de paso estándar estará bien para usar, pero a veces quieres un poco más de control sobre qué código se ejecuta y en qué momento, así que para eso se proporcionan los eventos de paso Begin y End . Los tres se comprueban en cada paso, pero su orden nunca variará incluso si las futuras actualizaciones del motor de GameMaker cambian otros eventos, lo que significa que este es el único método fiable para asegurarse de que algo siempre sucede antes que otra cosa.

¿Para qué se puede utilizar el evento de paso? Bueno, se puede utilizar para acciones o código que necesita ser ejecutado continuamente. Por ejemplo, si un object debe seguir a otro, aquí puedes adaptar la dirección del movimiento hacia el object que estamos siguiendo para que se mueva suavemente detrás. Ten cuidado con este evento, sin embargo, y no pongas muchas acciones complicadas en el evento de paso de objects, especialmente si estás planeando tener muchas instancias de object en tu juego room, ya que esto podría ralentizar el juego. Muchas cosas pueden ser colocadas en alarmas, o configuradas para que se disparen usando algunos de los Otros eventos, en lugar de que sucedan todo el tiempo.

Colisión

Obviamente, al hacer un juego, es muy importante que sepas cuando dos (o más) instancias de un object han colisionado, y para ello tenemos el Evento de Colisión. Este es un evento que se coloca en un object y luego se especifica contra qué otro object se debe comprobar si hay colisiones.

Si no tiene la física activada, estas colisiones se calcularán en base a la máscara de los dos objects (la máscara se define dentro de las propiedades de sprite, o se puede asignar independientemente en las propiedades de object ) y si se solapan o no. Ten en cuenta que si una de las instancias de la colisión no tiene asignada una máscara (o la máscara de sprite no está definida), aunque esté dibujando algo no se detectará ninguna colisión.

Si tiene activada la física , la colisión se basará en el tipo de forma de colisión(Fixture) que haya definido para el object en sus propiedades de física, al igual que su reacción a la colisión. Esto significa que puede no necesitar ningún código para tratar la colisión, pero este evento todavía necesitará tener al menos un comentario en él para que las colisiones sean detectadas.

Por último, hay que tener en cuenta que todas las colisiones se calcularán una vez por paso de juego antes de que se dispare el evento de colisión, de forma que cuando se ejecute el evento de colisión, todas las colisiones ya se habrán calculado y preasignado. Esto significa que si creas una instancia en este evento y luego intentas comprobar una colisión con ella, la colisión no será detectada o resuelta hasta la siguiente iteración del juego loop.

Teclado, Prensa del Teclado, Liberación del teclado

Dejar que el jugador controle los diferentes aspectos de su juego es muy importante, y para ello GameMaker le proporciona una lista muy completa de eventos de teclado que pueden ser utilizados en cualquiera de las tres categorías principales de teclado. Para la categoría general de Teclado, se dispara continuamente cada paso mientras la tecla seleccionada esté presionada, mientras que los eventos de la categoría Presionar y Soltar sólo se dispararán una vez cuando la tecla sea inicialmente presionada o liberada.

Hay que tener en cuenta que los eventos de teclado se activan en todas las instancias activas en un room cada vez que se utiliza una tecla, pero sólo responderán las que tengan un evento definido para esa tecla en particular y se pueden crear múltiples eventos de teclado en cualquier object y las instancias de ese object responderán a todos ellos mientras el juego esté en marcha.

Cuando añada cualquier evento de teclado a un object, se le presentará el menú de sub-evento de teclado donde puede especificar la tecla que va a comprobar:

La mayoría de ellos son bastante obvios, pero vamos a repasar brevemente las secciones - en la parte superior tenemos las teclas de flecha, seguidas por las teclas modificadoras más usadas, luego el resto del teclado (dividido en más sub-secciones para que puedas obtener la tecla exacta requerida como o ) y finalmente dos sub-eventos muy especiales, No Key y Any Key. Como sus nombres implican, estos son sub eventos que comprueban cuando no se presiona ninguna tecla o cuando se presiona cualquier tecla. Tenga en cuenta que las teclas del teclado numérico sólo producen los eventos correspondientes cuando el Bloqueo Numérico está activado.

Los eventos de Pulsar y Soltar para el teclado son casi exactamente los mismos que el evento regular de teclado, excepto que en lugar de activarse continuamente, se activan una sola vez. Cuando el teclado registra por primera vez que una tecla ha sido presionada, generará un evento de Teclado Presionado (así como un evento de Teclado regular), y la primera vez después de eso donde una tecla ya no es detectada como presionada, disparará un único evento de Liberación de Teclado.

Ratón

La categoría Ratón está separada en una serie de eventos que pueden ser seleccionados para darle un control más preciso sobre lo que está sucediendo en su juego. Aquí puedes ver exactamente cuáles son estos eventos:

Los eventos de los botones izquierdo , derecho y medio (ya sea normal, pulsado o liberado) funcionan todos en la máscara de la instancia que tiene el evento. Lo que esto significa es que GameMaker comprobará la posición del ratón en el room cuando se usen esos botones contra las máscaras de colisión de las instancias que tienen un evento de ratón. Si hay una "colisión" con el cuadro delimitador de la instancia, entonces el evento se activará, así que asegúrese de que cualquier instancia con estos eventos tiene un sprite con una máscara de colisión válida o que el object tiene una máscara sprite seleccionada en las propiedades de object. Como sus nombres implican, estos eventos se dispararán o bien una vez cuando se pulse o suelte por primera vez el botón del ratón elegido, o bien de forma continua en cada paso mientras se mantenga el botón.

Los eventos de entrada y salida del ratón también son similares a los eventos de botón en el sentido de que también dependen de la máscara de la instancia para funcionar, pero esta vez se activan cuando el ratón "entra" (toca) por primera vez la instancia o cuando el ratón "sale" (deja de tocar) la instancia. Sin embargo, estos eventos no son continuos, y se activan sólo una vez por cada vez que el ratón entra o sale de object - por lo que son un método ideal para crear, por ejemplo, botones que necesitan cambiar cuando el ratón pasa por encima de ellos antes de volver a la normalidad cuando el ratón se retira.

Finalmente tenemos otra sección para los eventos del ratón que se llama Ratón Global. En este submenú encontrarás una selección de eventos que sirven para registrar los eventos del ratón en las instancias aunque el ratón no esté sobre ellas o incluso cerca de ellas. Estos son eventos que se generan para todas las instancias y si hay acciones o código definido para el evento especificado entonces se ejecutará, independientemente de la posición del ratón dentro del juego room.

Tenga en cuenta que en los dispositivos móviles o de pantalla táctil el botón izquierdo del ratón también se puede utilizar para comprobar si hay una pestaña de dedo en una pantalla táctil, y el botón derecho del ratón se activa con un doble toque en la pantalla (este comportamiento se puede cambiar mediante un código).

Gestos

Este evento es el que se activará cuando el usuario toque la pantalla (en el móvil) o haga clic y mueva el ratón (en el resto de plataformas). Estos eventos son similares a los eventos del ratón, en el sentido de que tienes versiones regulares y versiones globales. Las versiones regulares de estos eventos sólo se activarán cuando los toques se produzcan en una instancia que tenga un sprite (o una máscara) y los toques se produzcan dentro de su caja de delimitación. Las versiones globales de estos eventos, sin embargo, se activarán cuando el usuario toque cualquier parte de la pantalla.

Los eventos gestuales detectan lo siguiente:

  • toques: cuando un usuario hace clic/toca y suelta rápidamente
  • arrastre - cuando un usuario toca/clica y lo mantiene y luego mueve su dedo/cursor
  • flicks - cuando el usuario mueve y suelta un toque/clic en un solo movimiento
  • pellizcos: cuando el usuario tiene dos dedos en la pantalla y los junta/separa
  • gira: cuando el usuario tiene dos dedos en la pantalla y los hace girar alrededor de un punto

Los diferentes eventos contendrán siempre un mapa DS llamado "event_data", que contendrá una serie de pares clave/valor con datos sobre la posición y el movimiento del toque/clic. Dibujar

Esta categoría de eventos es la que rige lo que se ve en la pantalla cuando se ejecuta el juego, y se divide en varios eventos discretos:

Como puede ver, la categoría de eventos de dibujo tiene varios tipos de eventos diferentes. Draw Begin, Draw y Draw End son los eventos de dibujo "estándar" que probablemente utilizarás más. Por defecto el evento principal Draw es siempre llamado para cada instancia, sin importar si tiene un sprite o no, aunque si usted flag la instancia como invisible, el evento no será disparado (así que tenga esto en mente si tiene alguna lógica de juego en el evento draw de un invisible object, ya que no se ejecutará). El evento de dibujo principal es también donde GameMaker dibuja por defecto la instancia sprite cuando no hay código ni acciones en el evento (es decir: no lo has añadido a la lista de eventos para el object). El dibujo por defecto utiliza el sprite asociado con la instancia y dibujará eso con cualquier transformación establecida en el código o acciones aplicadas.

Los eventos de dibujado estándar dibujan antes que los eventos de dibujado de la interfaz gráfica de usuario y entre los eventos de pre-dibujado y post-dibujado , lo que significa que todo lo que se dibuja en este evento se dibuja debajo del evento de dibujado de la interfaz gráfica de usuario , independientemente de la capa (es decir, cualquier cosa dibujada en el evento de dibujado de la interfaz gráfica de usuario siempre se dibujará sobre cualquier cosa dibujada en el evento de dibujado normal, independientemente del orden de las capas).
