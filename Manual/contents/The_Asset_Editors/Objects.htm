El editor de Object

Los objetos son el recurso que utilizamos para controlar aspectos de un juego y para hacer cosas específicas. La mayoría de las veces tienen un sprite asociado a ellos para que los veas en el juego rooms, pero a veces se utilizan como un controlador "entre bastidores" para hacer cosas relacionadas con el usuario o para la sincronización, etc. Se les puede dar comportamientos y pueden reaccionar a ciertos eventos, así como entre ellos, y la mayoría de las cosas que se ven en un juego están basadas en objects y sus interacciones. Ahora bien, decimos "basadas en" porque en realidad no colocas objects directamente en el juego rooms, sino que colocas instancias de estos objects que son básicamente copias (o clones si lo prefieres) del recurso object. Esto es algo muy importante que hay que recordar, ya que las instancias y objects no son lo mismo y cada una tiene su propio conjunto de funciones que pueden afectarlas.

Para aclarar esto, piensa en un juego hecho con GameMaker. Todos los personajes, monstruos, bolas, paredes, etc... son todos objects que usted crea en el Navegador de Activos. A continuación, arrastra el objeto en un room en el Editor de Salas para crear una instancia copiada de la plantilla básica object. La instancia en el room puede entonces ser cambiada en el propio editor o a través de código, de manera que puede ser escalada, o coloreada, o tener otros detalles modificados de la "base" object. Por tanto, cuando hablamos de que algo afecta o cambia una instancia, nos referimos a que afecta a esa copia concreta de un object en una sala. Sólo esa instancia se ve afectada, mientras que el resto no. Pero, cuando hablamos de afectar o cambiar un objeto, queremos decir que estamos modificando el object en el Navegador Asset y cualquier cosa que hagamos en él se reflejará también en todas las instancias creadas a partir de ese momento. Así, el object es la plantilla para la instancia, y la instancia es lo que colocamos en un room para hacer nuestro juego.

Para crear un object en su juego, puede hacer clic con el botón derecho del ratón en en el navegador Asset y hacer clic en la opción Crear objeto. Esto abrirá la ventana que aparece en la parte superior de esta página con las siguientes secciones:

Detalles

Aquí puede dar un nombre a su object. Este nombre nunca debe contener espacios ni caracteres no autorizados (sólo se permiten caracteres alfanuméricos y la barra inferior "_") y debe ser corto pero memorable para que pueda identificar fácilmente los recursos más adelante, cuando tenga muchos de ellos. Por ejemplo, mucha gente utiliza un prefijo o un sufijo para identificar las diferencias entre los recursos, como "obj_Ship" o "Ship_obj".

A continuación puede elegir el sprite para el object si desea que tenga uno. Haga clic con el botón izquierdo del ratón en la casilla sprite y aparecerá el Explorador de Activos con una lista de todos los sprites disponibles desde el Navegador Asset:

A continuación, puede seleccionar el que desee utilizar para el object. Con un sprite asignado al object puede hacer clic en el botón Editar Spr ite para cambiar las propiedades del sprite, o -si no tiene un sprite para el object pero desea crear uno- puede hacer clic en el botón Añadir Sprite para crear un nuevo sprite asset y asignarlo. También puede editar la imagen de sprite asset que ha sido asignada asignada haciendo clic en el botón Editar imagen para abrir el Editor de imágenes.

Máscara de colisión

Debajo del área de asignación de sprite tiene un botón para elegir una máscara de colisión específica. Por defecto, un object generará propiedades de colisión basadas en el sprite que se le ha asignado, pero hay casos en los que puede querer colisiones se basen en una forma diferente de un sprite diferente. Por ejemplo, puedes tener un juego de arriba a abajo donde el sprite principal tiene una máscara de colisión rectangular, pero para hacer las colisiones más suaves una máscara circular puede ser mejor. En estos casos puede hacer clic en el botón de máscara de colisión y seleccionar otro recurso sprite, de modo que ahora las instancias de object dibujarán el sprite asignado, pero basarán las colisiones en la máscara sprite. Puedes encontrar más información sobre las máscaras de colisión en la página del Editor de Sprite.

Propiedades

Aquí se le presentan las siguientes opciones para su objeto:

  • Visible: Visible indica si las instancias de este object son visibles cuando se inicia el room. Normalmente la mayoría de las instancias son visibles, pero a veces es útil tener instancias invisibles - por ejemplo, puedes usarlas como puntos de paso para controlar un monstruo en movimiento, o para mantener valores de seguimiento y realizar ciertas acciones cronometradas. Los objects invisibles seguirán reaccionando a los eventos y si tienen un sprite o una máscara asignada, también reaccionarán si otras instancias colisionan con ellos... simplemente no pueden ser vistos y no realizan su Evento de Dibujo. Por defecto, la opción visible flag está marcada.
  • Sólido: Cuando usted flag una instancia como sólida, le está diciendo a GameMaker que debe tratar de resolver cualquier colisión antes de disparar un Evento de Colisión. Esta es una funcionalidad muy básica, y en realidad todo lo que hace es mover la instancia de vuelta a la posición en la que estaba en el momento antes de que se detectara una colisión antes de realizar cualquier código o GML Visual que está en el propio Evento de Colisión, y como tal es de uso limitado.
  • Persistente: Un object persistente es aquel que no desaparece cuando se cambia el room, sino que "persiste" y se traslada al nuevo room. Sólo desaparecerá cuando lo destruyas explícitamente, ya sea con una acción de destrucción o en código. Esto significa que si colocas un object persistente en un room, seguirá estando disponible en todos los rooms subsiguientes y continuará realizando cualquier acción asignada a los eventos en sus propiedades object, excepto el Evento Crear, que sólo se dispara una vez cuando se crea inicialmente y no cuando se cambia a un nuevo room. Esto es útil para cuando tienes, por ejemplo, un personaje principal que se mueve de room a room y quieres mantener las variables dentro de la instancia igual, pero debes tener mucho cuidado de crear y destruir correctamente objects persistente, ya que puedes conseguir fácilmente errores que se cuelen en tu proyecto por error.

Persistencia y Room Capas

Cuando se crea una instancia con persistencia habilitada en un room, se le asignará una capa o una profundidad (dependiendo de la función que se haya utilizado para crear la instancia del object). Cuando se cambia el room, y si el siguiente room no tiene una capa con el mismo nombre o profundidad que la asignada, entonces se creará una nueva capa para la instancia que persiste a través de las salas. Si usted asignó una capa con nombre cuando la instancia fue creada, entonces la nueva capa también tendrá el mismo nombre que la capa original, mientras que si usted asignó una profundidad a la instancia entonces la nueva capa se llamará "_layer_XXX", donde "XXX" es un valor hexadecimal utilizado para dar a la capa un nombre único. Estas capas se eliminarán de la página room al final de la sala.

Otra cosa a tener en cuenta sobre las capas y las instancias persistentes es que si has asignado a la instancia una capa con nombre en la creación, y hay otra capa en la siguiente rooms con el mismo nombre, entonces la instancia persistente será asignada a la capa con el mismo nombre independientemente de la profundidad de la capa.

Por último, si una instancia persistente se mueve a un room con una capa con la misma profundidad en la que se creó la instancia, no se asignará a esta capa, sino que se creará una nueva capa con la misma profundidad (siguiendo la convención de nombres explicada anteriormente).

Persistencia y eventos

Como se mencionó anteriormente, un object persistente sólo ejecutará su Evento de Creación una vez, pero también tendrá sus eventos de Inicio de Juego, Fin de Juego, Inicio de Sala y Fin de Sala disparados en los momentos apropiados. Sin embargo, si reinicias el juego (con, por ejemplo, la función game_restart() ) todos los objects persistentes serán eliminados y sólo existirán cuando sean creados de nuevo por el juego.

Ten en cuenta también que si desactivas un object persistente, ya no pasará de un room a otro a no ser que se reactive antes de que se active el evento Room End. La reactivación requiere un paso completo del juego para que ocurra, por lo que usar el evento Room End para esto no funciona, ya que no le da al juego un paso completo para reactivar el object, por lo que debes tener esto en cuenta cuando uses la desactivación junto con el objects persistente y planificar tu código en consecuencia.

  • Usa Física: Cuando usted marca la casilla Uses Physics le está diciendo a GameMaker que este object debe ser parte de un mundo de física (como se establece en el Room Editor. Al marcar esta casilla también se abrirá una ventana encadenada adicional donde puede definir las propiedades físicas de todas las instancias de este object. Más información sobre esto se puede encontrar en la sección de Objetos de Física.
  • Gestionado: Si este object debe ser gestionado cuando se utiliza Rollback Multiplayer. El estado de un objeto gestionado puede ser revertido y se pueden ejecutar predicciones sobre él. Ver Objetos Gestionados.
